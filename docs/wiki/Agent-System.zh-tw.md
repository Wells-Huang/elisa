# 代理系統 (Agent System)

Elisa 的後端協調 AI 代理來構建軟體與硬體專案。本頁面涵蓋代理角色、提示詞、執行、上下文共享、權限以及重試機制。

## 代理角色 (Agent Roles)

| 角色 | 用途 | 提示詞範本 |
|------|---------|----------------|
| **構建者 (Builder)** | 撰寫原始碼 | `backend/src/prompts/builderAgent.ts` |
| **測試者 (Tester)** | 撰寫並執行測試 | `backend/src/prompts/testerAgent.ts` |
| **評閱者 (Reviewer)** | 評審程式碼品質 | `backend/src/prompts/reviewerAgent.ts` |
| **自定義 (Custom)** | 使用者定義的個性 | 以構建者提示詞為基礎 |
| **講述者 (Narrator)** | 將事件轉化為適合兒童的評論 | `backend/src/prompts/narratorAgent.ts` |

代理是透過畫布上的僕從 (Minion) 積木進行配置的。若未放置僕從積木，Elisa 將使用預設團隊（一位構建者、一位測試者）。

## 代理提示詞 (Agent Prompts)

每位代理都會收到一個由以下內容構成的系統提示詞：

1. **角色範本** — 代理角色的基礎指令（構建者、測試者、評閱者）
2. **個性 (Persona)** — 孩子對代理個性的描述（來自僕從積木）
3. **任務說明** — 代理需要做什麼，以及驗收標準
4. **前序任務上下文** — 先前代理所做工作的摘要（上下文鏈）
5. **檔案清單** — 工作區中存在哪些檔案
6. **風格需求** — 來自風格積木的視覺與個性設定
7. **技能** — 適用於此任務的啟用中技能提示詞
8. **規則** — 根據觸發類型啟用的規則提示詞
9. **內容安全** — 強制執行適合年齡的產出 (適合 8–14 歲)

### 構建者提示詞重點

- 撰寫初學者也能理解的、簡潔且具備良好註釋的程式碼
- 使用簡單的變數命名
- 優先考慮清晰度而非花俏技巧
- 優雅地下處理錯誤
- 撰寫適合兒童的 Git 提交訊息
- 尊重檔案邊界

### 測試者提示詞重點

- 每個測試只驗證一件事
- 同時測試正常路徑與邊緣案例
- 以通過/失敗指標來回報結果
- 適用時包含測試覆蓋率

### 評閱者提示詞重點

- 檢查驗收標準、易讀性、臭蟲、簡潔度
- 提供結論：已批准 (APPROVED) 或 需要修改 (NEEDS CHANGES)
- 給予鼓勵 — 指出優點，而不僅僅是問題點

## 執行模型 (Execution Model)

每個代理任務都透過 `AgentRunner` 作為獨立的 Claude Agent SDK `query()` 調用運行：

```
AgentRunner.execute(prompt, options)
  → 調用 SDK query() 並設定 permissionMode: 'bypassPermissions', maxTurns: 20
  → 透過非同步疊代串流傳輸助理訊息
  → 提取結果中繼資料 (Token, 成本)
  → 回傳結果摘要
```

**配置：**
- 超時：每位代理 300 秒
- 重試：失敗時最多重試 2 次
- 模型：`claude-opus-4-6` (可透過 `CLAUDE_MODEL` 配置)
- 最大輪次：20 輪

## 串流並行執行 (Streaming-Parallel Execution)

協調器 (Orchestrator) 使用串流並行池從 DAG 中執行任務：

1. **Kahn 演算法** 對任務進行拓撲排序
2. 所有相依關係皆已滿足的任務進入「就緒」佇列
3. 透過 `Promise.race` 同時運行最多 **3 個任務**
4. 當任一任務完成時，下一個就緒任務即刻開始
5. Git 提交透過互斥鎖序列化（一次僅能進行一個提交）
6. 每次任務完成後都會檢查 Token 預算

```
就緒佇列：[task_1, task_2, task_3, task_4, ...]
執行池 (最多 3 個)：[task_1 ⚙️, task_2 ⚙️, task_3 ⚙️]
                        ↓ task_1 完成
執行池：              [task_4 ⚙️, task_2 ⚙️, task_3 ⚙️]
```

## 上下文鏈 (Context Chain)

每個任務結束後，會寫下一個上下文摘要，以便後續代理了解發生了什麼事：

1. 代理完成任務並產生輸出
2. 從輸出中提取一個 **摘要**
3. 產生目前工作區檔案的 **結構化摘要 (structural digest)**
4. 將兩者寫入 `.elisa/context/nugget_context.md`
5. 下一個代理在其提示詞中的「在你之前的進度」區塊收到此上下文

### 上下文層級

| 層級 | 何時 | 提供的上下文 |
|-------|------|-----------------|
| 1 | 骨架構建代理 | 僅專案規格 |
| 2 | 首批實作代理 | 規格 + 骨架摘要 + 檔案清單 |
| 3 | 具相依關係之代理 | 規格 + 所有前序摘要 + 檔案清單 |
| 4 | 整合/測試代理 | 以上所有內容 + 評核回饋 |
| 5 | 部署代理 | 以上所有內容 + 測試結果 + 建置產物 |

## 權限原則 (Permission Policy)

`PermissionPolicy` 根據可配置的規則自動解析代理的權限請求：

| 權限類型 | 策略 |
|----------------|--------|
| 檔案寫入 (在工作區內) | 自動批准 |
| 檔案寫入 (在工作區外) | 拒絕 |
| 唯讀指令 (`ls`, `cat`, `grep`) | 始終安全 |
| 工作區指令 (`mkdir`, `python`, `npm`) | 若 cwd 在工作區內則批准 |
| 網路指令 (`curl`, `wget`) | 預設拒絕 |
| 套件安裝 (`pip install`, `npm install`) | 升級呈報給使用者 |

每個任務若權限被拒絕達 3 次，策略會將權限請求呈報給使用者。

## 任務分解 (元規劃器 MetaPlanner)

元規劃器呼叫 Claude (Opus 模型) 將 NuggetSpec 分解為任務 DAG：

1. 接收完整的 NuggetSpec 作為輸入
2. 套用分解規則（4–12 個任務，每個 1–5 分鐘，無循環）
3. 回傳結構化 JSON：包含相依關係、驗收標準及代理分配的任務
4. 驗證 DAG 是否有循環
5. 若 JSON 解析失敗則重試

### 分解規則

- 第一個任務始終是專案骨架 (scaffolding)
- 最後一個任務是整合驗證
- 每個任務都有明確的驗收標準
- 硬體任務包含編譯驗證
- 構建者永遠不應評閱自己的工作
- 相依關係遵循邏輯順序 (骨架 → 實作 → 測試 → 整合 → 部署)

## 教學引擎 (Teaching Engine)

教學引擎在建置過程中呈現情境化的學習時刻：

1. **快速路徑查詢**：將建置事件映射至預定義概念的課程中
2. **去重**：每個概念在一個工作階段中僅顯示一次
3. **API 備案**：當無法匹配時，退而使用 Claude Sonnet 進行動態產生
4. **目標受眾**：8–14 歲

涵蓋概念：原始碼控制、測試、分解、程式碼評閱、架構、硬體 (GPIO, LoRa, 編譯, 燒錄)。

## 講述者服務 (Narrator Service)

講述者服務將原始建置事件翻譯為適合兒童的評論：

- 使用 Claude Haiku (`NARRATOR_MODEL` 環境變數)
- 4 種情緒：興奮、鼓勵、擔憂、慶祝
- 頻率限制：每個任務每 15 秒最多 1 條訊息
- `agent_output` 事件去抖動 (10 秒靜默期)
- API 超時時使用預設範本作為備案

## 錯誤處理與重試

| 情境 | 行為 |
|----------|----------|
| 代理超時 (>300s) | 使用相同提示詞重試 (最多 2 次) |
| 代理產生無效輸出 | 將錯誤回饋給代理進行自我修正 |
| DAG 中存在循環相依 | 在規劃階段予以拒絕 |
| 超出 Token 預算 | 發出警告並優雅停止 |
| 重試次數耗盡 | 人類閘門：「這部分我們遇到了一些困難」 |
| 硬體編譯失敗 | 向使用者呈現錯誤細節 |
| 燒錄超時 (>60s) | 最多重試 3 次 |

## 取消 (Cancellation)

當建置被取消（按下 STOP 按鈕或發生錯誤）時：

1. `Orchestrator.cancel()` 觸發 AbortController
2. 信號傳播至所有活動中的 SDK `query()` 調用
3. 代理立即中止
4. 工作階段狀態設為 `done`
5. 清理 WebSocket 連線
